mocha/chaiで爆速TDD開発【node】

アドベントカレンダーです

  • めっちゃ申し訳ないですが、複数のカレンダーに同時に掲載する気まんまんです
    • (まったく申し訳ないと思ってない)
  • もう年末なんですね

本題

テスト駆動開発しようぜ

  • テストから書いて実装しようぜってやつです
  • いろいろ意見はあると思うけど
    • 実装前後構わずテストは書こうぜって意見を持ってる
  • その上でテスト駆動にしちゃえば要件からズレずに実装できるじゃん
    • そんな気持ちでテスト駆動開発してます

Node.jsで書いたウェブアプリでテスト駆動

  • リクエストレスポンスがあるウェブ系の開発でテスト駆動難しそう
    • でもそんなことないよ！
    • npm installだけで爆速で環境設定できちゃう

環境

フォルダ構成

  • こんな感じでエンドポイントがある階層にtestディレクトリを作ります
    • 別にこんな感じにしなくてもいいけど、test配下とrootでファイルが1:1対応させといた方が良い

インストール

  • 全部npmで入っちゃう!すごい!

  • テスティングライブラリ

    • $ npm install chai
    • npmjs.com/chai
    • アサーションライブラリってやつ
      • テストコードが書くのに特化したメソッドいっぱい持ってる
  • テスティングライブラリのHTTPリクエスト用プラグイン
    • $ npm install chai-http
    • npmjs.com/chai-http
    • chaiでhttpリクエスト/レスポンスできるようになるやつ
      • これさえあればエンドポイント叩く系のテストコードはなんでも書ける気がするよ！

  • テスティングフレームワーク

    • $ npm install mocha
    • npmjs.com/mocha
    • テスト結果をまとめてくれるすごいやつ
      • chaiとかのライブラリが出力したものをヒューマンリーダブルに出力してくれる

  • なんか他にも色々ライブラリやフレームワークはあるよ

    • 僕が感覚的に書けるのはこの組み合わせ
      • プロジェクトによって違ったりするから選択は難しいよね

TDD(テスト駆動開発)を始めるぞ！

  • 今回は数字を投げたらそれが10と比較した結果を返してくれるエンドポイントを作るよ！
  • テスト駆動だからテストから書いていく
    • コツはスペックを書いていくこと
      • 「このエンドポイントはこんな風に動いてほしい！」
      • そんな欲望を書く、それだけ

./dev/ytm-test/test/index.test.js

  • ソースとchai/mochaのチュートリアル見ればすぐ書ける
  • 今回は8080ポートにあるエンドポイントに数字をpostしたら
    • それが10と比べてどんな数字なのかを返すエンドポイントを作るよ！
  • イメージしやすいようにcurlを貼っておく
  • ./dev/ytm-test/test/index.test.jsは上のcurlをchai-httpの力を借りてリクエスト
  • chai-httpの力を借りてレスポンスを受け取って
    • chaiの力を借りてレスポンスと期待値が一緒であることを確認
  • mochaの力を借りてdescribeとitを使ってヒューマンリーダブルな出力がされるようにしているよ！

とりあえず実行してみる

  • とりあえずエンドポイント作る
    • express使ってテキトーなエンドポイント作ってプロセス起動しとく
  • mochaはnpm installでインストールしてるのでnode_modules以下からに実行が爆速かも
  • $ ./node_modules/mocha/bin/mocha test/index.test.js
  • スペックを全然満たしてないことがわかる
    • リクエスト受けたらとりあえずjsonでなんもないよ！を返すだけだからね
  • mochaはこんな風に
    • どのテストケースの
    • どのテストに
    • どうして失敗したのか
      • を綺麗に表示してくれます

開発開始

  • テストコードが書けたところでindex.jsがスペックを満たすように書いていきます
  • とはいえ簡単なエンドポイントなので一瞬で書いちゃいましょう

./dev/ytm-test/index.js

  • 単純明快ですね
    • 受け取った数字をifで評価してresultに結果を代入して返します
  • さっそくmochaでテストしてみましょう！
  • あれれ？ケース2が失敗してますね。
  • ./dev/ytm-test/index.jsのコードを確認してみましょう

./dev/ytm-test/index.js#L16

  • めちゃくちゃ初歩的なミス発見
  • 条件分岐を「10以下」、にしたいのに「10未満」になっていますね
    • これは境界値分析と言って、境界値を調べるテストケースで発見できるバグです
  • 直して再度mochaを実行してみよう
  • やった！全部のスペックを満たしたぞ！

別にテスト駆動開発は怖くない

  • 以上で、テスト駆動開発の爆速チュートリアルは終わりです
  • chaiにはexpect以外にも便利なメソッドがたくさん用意されています
  • 新規開発するならぜひテスト駆動してみてください

なんでテスト駆動なの？

  • 僕の意見は
    • 自分のコードを責任転嫁から守るためです
  • 複数人で開発してるとき、誰がそのコードを動かなくしたかわからなくなりがちです
  • でもコミットごとにテストコードを実行すれば、そのコミットを作ったやつが犯人です
  • 他人が直接コードをいじってなくても、依存関係にあるファイルをいじったせいで自分のコードが動かなくなることもあります
    • テストコードがあればそんなときも安心
    • ちゃんとスペック満たしたコードであることを胸を張って言える
    • 動かなくなったのは自分のコードではなく、あなたのコードが悪い
  • 堂々と主張できるように、ちゃんとテストコード書こうな！

付録

  • マークダウンだと全然mochaのヒューマンリーダビリティが伝わらなかったのでスクショつけときます

mochaにおけるdescribeのベストプラクティス

mochaのベストプラクティス

Node.jsのテスティングフレームワークであるmochaで困ったことがあったので解決策の覚書

起きたこと

  • テスト用にデータベースを作って、開発メンバーで共有して使っている
  • そのデータベースをいじって色々テストする
    • もちろん他のテストに影響を与えないように初期化しながら使う
  • あるとき、何もテストが走っていない状態でそのデータベースが初期化されていなかった

原因

  • テストが終わったあとにデータベースをinitializeできていなかった
  • beforeEachを使っていたが、afterはなかった

解決策

  • beforeEachだけだとitの直前のタイミングでしかinitializeが走らない
  • テストが終わっても最後のitのあとにデータベースにレコードが残っちゃう

  • afterも組み合わせてdescribeが終わったら綺麗にしてから帰りましょう、というお話

  • 往々にして、テストケースの最後は異常系とかにしてしまう

    • データベースに更新がかからないテストケースなので、afterしなくても平気な場合が大半
    • でも明示的にinitializeして気持ちよくテストできるようにしようね

beforEachのみで
テストケースの最後に異常系を置いたつもりが正常にデータベースが更新されていてテストが落ちた
みたいなことが起きると事件になります
気をつけましょう(経験談)

VSCode_win10からコミットする

win10からやるならコマンドはgitしか使いたくない人向け。

VSCode_win10でGitHubにえいってする

Git Bash

  1. $ git config --global user.name 'ユーザー名'
  2. $ git config --global user.email 'メールアドレス@gmail.com'

Git Gui

  1. Help > Show SSH Key
  2. Generate Key

ブラウザ(GitHub)

  1. SSH Keyの登録

VSCode

  1. $ git init
  2. $ git remote add origin https://github.com/***/***.git
  3. コマンドラインorソース管理メニューでadd commit push

以上。
gitだけはターミナルでやった方が確実。
